# 第08堂｜實作：智慧多通路內容行銷與分發系統

## 學習目標

完成本堂課後，學員將能夠：

- 設計並封裝 AI 代理的指令集，讓 Agent 能可靠地執行內容生成任務
- 使用 Prompt Engineering 技巧，讓 AI 針對不同平台產出適合的行銷內容
- 建立自動化的多平台內容分發工作流
- 了解各平台 API 的串接方式與使用限制

---

## 核心概念說明

### 1. 多通路內容行銷的挑戰

現代行銷需要同時在多個平台上發布內容：

| 平台 | 限制與特性 |
|------|-----------|
| Facebook | 長文適合，支援圖片、影片 |
| Instagram | 視覺優先，需高品質圖片，文字相對次要 |
| LinkedIn | 專業語氣，適合 B2B 內容，長文效果好 |
| X（Twitter） | 280 字元限制，短而有力 |
| LINE 官方帳號 | 台灣本地化，需要本地語氣，支援彈性選單 |
| 電子郵件電子報 | 可包含詳細資訊，需要吸引人的主旨 |

**手動管理多平台的問題：**
- 每個平台需要重新改寫適合的版本，費時費力
- 容易發布風格不一致的內容
- 人工失誤（忘記發佈、發錯版本）頻率高

---

### 2. Prompt Engineering 用於內容生成

**Prompt Engineering（提示詞工程）**是設計和優化給 AI 的指令，讓 AI 產出更準確、更符合需求的輸出。

**控制輸出品質的關鍵 Prompt 要素：**

| 元素 | 說明 | 範例 |
|------|------|------|
| **角色設定** | 告訴 AI 它扮演什麼角色 | 「你是一位有 10 年經驗的品牌行銷專家」 |
| **目標受眾** | 描述內容的目標讀者 | 「目標受眾是 30-40 歲的台灣中階主管」 |
| **平台特性** | 說明發布平台的規則和風格 | 「這是 LinkedIn 貼文，語氣要專業，可以長一點」 |
| **內容要素** | 指定必須包含的資訊 | 「必須包含產品名稱、三個主要優點、行動號召」 |
| **格式要求** | 指定輸出的格式 | 「字數限制 200 字，要有 3 個 hashtag」 |
| **語氣規範** | 描述希望的寫作風格 | 「語氣要活潑友善，不要太制式」 |

**針對不同平台的內容改寫策略：**

```
# 輸入：一份產品資訊文件
產品名稱 = "SmartFlow AI 方案"
核心賣點 = ["節省 70% 人工時間", "無需寫程式", "3 分鐘上手"]
目標行動 = "立即申請免費試用"

# 讓 AI 改寫為各平台版本：

Facebook 版：
    語氣友善、故事化，字數 150-300 字，加入 emoji

LinkedIn 版：
    專業商務語氣，說明商業價值，字數 200-400 字，
    包含數據證明，加入 2-3 個相關 hashtag

X 版：
    濃縮核心訊息，280 字元限制，1 個強烈的 CTA

Email 主旨：
    吸引人點擊、製造好奇心，50 字元以內
```

### 3. AI 指令集封裝（Function/Tool 設計）

當 Agent 執行複雜任務時，需要將整個工作流拆解為**可重複使用的工具函式**，再由 Agent 根據任務需求選擇調用哪些工具。

**內容行銷 Agent 的工具清單設計：**

```
工具 1：生成平台內容(產品資訊, 平台名稱, 語氣)
    → 輸出: 該平台的貼文草稿

工具 2：調整內容長度(文字, 目標字數)
    → 輸出: 調整後的文字

工具 3：加入 Hashtag(文字, 數量, 主題類別)
    → 輸出: 加上 hashtag 的文字

工具 4：生成圖片描述(產品資訊, 平台風格)
    → 輸出: 適合的圖片描述（可交給設計師或 AI 生圖工具）

工具 5：排程發布(平台, 內容, 發布時間)
    → 觸發對應平台 API 進行發布

工具 6：記錄發布結果(平台, 發布時間, 成功/失敗)
    → 儲存到發布記錄 Excel
```

### 4. 平台 API 串接概念

各社群平台都提供官方 API 供開發者使用，但各自有不同的規範：

**常用平台 API 概覽：**

| 平台 | API 名稱 | 申請難度 | 主要限制 |
|------|---------|---------|---------|
| Facebook/Instagram | Meta Graph API | 中（需審核） | 每日發文次數限制 |
| LinkedIn | LinkedIn Marketing API | 中（需申請） | 需要 Page 管理員權限 |
| X（Twitter） | X API v2 | 低（基本免費） | 免費版有讀寫限制 |
| LINE 官方帳號 | LINE Messaging API | 低 | 需要在台灣申請官帳 |

**API 串接的通用流程：**

```
1. 到平台的開發者中心申請帳號
2. 建立應用程式（App），取得 API Key 和 Secret
3. 完成 OAuth 2.0 授權流程，取得 Access Token
4. 使用 Access Token 呼叫 API 發布內容
5. 處理 API 的回應（成功/失敗/限流）
```

**自動排程發布的概念：**

```
觀念：不是立即發布，而是依照最佳時間排程

研究顯示的最佳發布時間（參考）：
    Facebook: 週三 11:00, 13:00
    Instagram: 週二-週五 10:00-14:00
    LinkedIn: 週二-週四 07:00-08:00, 17:00-18:00
    X: 週一-週五 09:00, 12:00, 17:00
```

## 實作步驟（逐步教學）

### 實作一：建立多平台內容生成引擎

**步驟 1：設計各平台的 Prompt 模板**

向 GitHub Copilot 描述：
```
請幫我建立一個「多平台內容生成器」，讀取一份產品資訊的 JSON，
並針對以下 4 個平台分別生成適合的貼文草稿：
1. Facebook（200-300字，親切風格，含 emoji，含行動號召）
2. LinkedIn（200-350字，專業商務語氣，含數據，2-3個hashtag）
3. X/Twitter（100字以內，精煉，含1個hashtag，1個行動號召）
4. 電子郵件主旨（50字以內，吸引人點擊）

輸入格式（JSON）：
{
    "product_name": "產品名稱",
    "features": ["特點1", "特點2", "特點3"],
    "target_audience": "目標受眾描述",
    "cta": "行動號召文字"
}

輸出格式：每個平台一個字典，包含「草稿」和「字數」
```

**步驟 2：測試生成品質**

準備 2-3 個不同類型的產品資訊作為測試輸入，評估 AI 生成的內容：
- 語氣是否符合平台風格？
- 字數是否在限制內？
- 是否包含所有必要資訊？

---

### 實作二：加入人工審核流程

**步驟 1：設計「草稿審核介面」**

```
請幫我建立一個互動式的草稿審核程式：
1. 依序顯示每個平台的內容草稿
2. 對使用者詢問：a) 核准發布 b) 修改後發布 c) 跳過
3. 如果選擇「修改」，讓使用者輸入修改需求，AI 重新生成
4. 將所有核准的草稿儲存到 approved_drafts.json 檔案
5. 生成一份「發布計畫表」Excel，記錄各平台、草稿內容、預計發布時間
```

---

### 實作三：串接 LINE Notify（示範 API 發布流程）

> 本實作以 LINE Notify 為範例（申請最簡單），示範 API 串接的完整流程。

**步驟 1：申請 LINE Notify Token**
1. 前往 [https://notify-bot.line.me/zh_TW/](https://notify-bot.line.me/zh_TW/)
2. 登入 LINE 帳號
3. 點選「個人頁面」→「發行存取權杖（開發人員用）」
4. 複製取得的 Token，設定為環境變數

**步驟 2：向 Copilot 描述 API 串接需求**

```
請幫我用 Python 寫一個函式，透過 LINE Notify API 發送訊息：
1. 函式名稱：send_line_notification(message)
2. LINE Notify Token 從環境變數 LINE_NOTIFY_TOKEN 讀取
3. 呼叫 https://notify-api.line.me/api/notify 傳送訊息
4. 加入錯誤處理：401 = Token無效、429 = 超出限制，分別給出明確的提示
5. 發送成功後回傳 True，失敗回傳 False
```

## 程式碼範例（虛擬碼）

### 多平台內容生成函式

```
函式 生成多平台內容(產品資訊):

    平台設定 = {
        "Facebook": {
            "字數限制": "200-300字",
            "風格": "親切友善，含適量 emoji",
            "必要元素": "產品特點 + 故事/情境 + 行動號召"
        },
        "LinkedIn": {
            "字數限制": "200-350字",
            "風格": "專業商務語氣",
            "必要元素": "商業問題 → 解決方案 → 數據佐證 → CTA + hashtag"
        },
        "X": {
            "字數限制": "100字以內",
            "風格": "精煉有力",
            "必要元素": "核心訊息一句話 + 1個hashtag + 連結"
        },
        "Email主旨": {
            "字數限制": "50字以內",
            "風格": "製造好奇心或緊迫感",
            "必要元素": "主題清楚 + 引發點擊動機"
        }
    }

    結果 = {}

    對 每個平台, 設定 在 平台設定.items():
        提示詞 = f"""
        你是一位專業的行銷文案撰寫師。
        請為以下產品/服務撰寫 {平台} 的貼文：

        產品名稱：{產品資訊["product_name"]}
        主要特點：{產品資訊["features"]}
        目標受眾：{產品資訊["target_audience"]}
        行動號召：{產品資訊["cta"]}

        要求：
        - 字數：{設定["字數限制"]}
        - 語氣風格：{設定["風格"]}
        - 必須包含：{設定["必要元素"]}
        - 使用繁體中文

        只回傳貼文草稿，不要說明。
        """
        草稿 = 呼叫LLM(提示詞)
        結果[平台] = {
            "草稿": 草稿,
            "字數": 計算字數(草稿)
        }

    回傳 結果
```

### 發布計畫表生成

```
載入 pandas, openpyxl, datetime 套件

函式 生成發布計畫表(已核准草稿, 輸出檔案="發布計畫表.xlsx"):
    資料 = []
    發布時間計畫 = {
        "Facebook": "週三 11:00",
        "LinkedIn": "週二 08:00",
        "X": "週一-週五 12:00",
        "LINE": "週三 18:00"
    }

    對 每個平台, 內容 在 已核准草稿.items():
        資料.加入({
            "平台": 平台,
            "草稿內容": 內容["草稿"],
            "字數": 內容["字數"],
            "建議發布時間": 發布時間計畫.get(平台, "待定"),
            "狀態": "待發布",
            "發布者": "",
            "備註": ""
        })

    發布計畫 = 建立DataFrame(資料)
    發布計畫.寫入Excel(輸出檔案)
    顯示 f"發布計畫表已生成：{輸出檔案}"
```

## 練習題 / 作業

### 課堂練習

**練習 1：Prompt 差異測試**

使用同一份產品資訊，用以下三種不同的 Prompt 生成 Facebook 貼文，比較差異：
1. 只提供產品名稱和特點（最少資訊）
2. 加入目標受眾和語氣要求
3. 加入角色設定、目標受眾、語氣、字數、必要元素（完整 Prompt）

觀察：Prompt 越完整，輸出品質有何不同？

思考你的工作場景，選一個你想行銷的產品或服務，填寫：
