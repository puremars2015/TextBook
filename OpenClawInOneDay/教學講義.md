# OpenClaw 教學：從養第一隻小龍蝦到打造小龍蝦軍團

## 前言

OpenClaw 是一套可以安裝在自己電腦或伺服器上的 AI 助理系統。你可以把它想成一個「有大腦、有工具、可自動化、可接多平台訊息」的 AI Gateway。它不是單純聊天機器人，而是一個可以接上模型、技能、排程、訊息平台與自訂工作流程的個人 AI 中樞。

這份講義分成基礎教學與進階教學兩段。基礎段會帶你把 OpenClaw 安裝起來，知道怎麼啟動、怎麼對話、怎麼排錯。進階段則會帶你理解多 Agent、SubAgent、Skill、ClawHub、cron、heartbeat，最後把它們組合成一支真正能工作的「小龍蝦軍團」。

> Windows 使用者先記住一件事：官方文件目前仍偏推薦用 WSL2 跑 OpenClaw；但如果你的課程或工作環境就是原生 Windows，本教材也提供 Windows 原生流程與輔助安裝工具。

---

## 一、什麼是 OpenClaw？

OpenClaw 可以用一句話來定義：

**OpenClaw 是一個自架式、多通路、可擴充的 AI 助理平台。**

它的核心特色有五個：

1. **自架式**：資料與流程可以留在自己的電腦或伺服器，不一定要把工作內容交給第三方平台。
2. **多通路**：可以接 Telegram、Slack、Discord、Google Chat、LINE、WhatsApp 等多種訊息入口。
3. **可用工具**：不只是聊天，還能讀檔、執行指令、查資料、呼叫技能與自動化流程。
4. **多 Agent 架構**：可以讓不同 AI 分工，例如一個負責客服、一個負責研發、一個負責監控。
5. **可自動化**：支援 cron、heartbeat、webhook 等機制，讓 AI 主動幫你巡檢、提醒、彙整與回報。

如果你以前只用過 ChatGPT 網頁版，那 OpenClaw 的差別在於：

- ChatGPT 比較像「一個單點聊天介面」。
- OpenClaw 比較像「一套可以長期運作的 AI 作業平台」。

你可以把它理解成：

- **模型**是大腦
- **Gateway**是中控主機
- **Skill**是技能模組
- **Channel**是外部通訊管道
- **cron / heartbeat**是自動巡邏與排程機制
- **SubAgent / Multi-Agent**是分身與團隊分工

## 二、先備工作

### 2.1 需要的工具與軟體

要順利使用 OpenClaw，至少先準備以下項目：

1. **Node.js**
2. **Git**
3. **PowerShell 7**（Windows 建議）
4. **VS Code**（建議）
5. **GitHub Copilot**（建議，但不是 OpenClaw 的必要條件）
6. **一組可用的模型帳號或 API Key**

依照目前官方說明，執行環境建議如下：

- **Node.js 24**：官方推薦
- **Node.js 22.14+**：最低相容版本
- **Windows**：建議搭配 PowerShell 7，若要最貼近官方流程可考慮 WSL2

### 2.2 安裝 Node.js 與 Git

你可以直接從官網安裝，也可以在 Windows 用 `winget`。

#### 方法 A：官網安裝

- Node.js: https://nodejs.org/zh-tw/download
- Git: https://git-scm.com/install/windows

#### 方法 B：Windows 指令安裝

```powershell
winget install Microsoft.PowerShell
winget install OpenJS.NodeJS.LTS
winget install Git.Git
```

安裝完成後，可以在 PowerShell 驗證：

```powershell
pwsh --version
node --version
npm --version
git --version
```

### 2.3 安裝 GitHub Copilot

GitHub Copilot 不是 OpenClaw 的必要元件，但如果你會在 VS Code 中維護 Skill、設定檔、腳本、排程或教學內容，Copilot 會大幅提升編輯效率。

安裝方式：

1. 開啟 VS Code
2. 進入 Extensions
3. 搜尋並安裝：
	 - `GitHub Copilot`
	 - `GitHub Copilot Chat`
4. 安裝完成後按右上角登入 GitHub

### 2.4 申請 GitHub Copilot 帳號

如果你還沒有 GitHub Copilot：

1. 先註冊 GitHub 帳號
2. 到 GitHub Copilot 頁面選擇方案
3. 完成訂閱或試用
4. 回到 VS Code 登入同一個 GitHub 帳號

> 提醒：OpenClaw 本身不依賴 GitHub Copilot 才能啟動。Copilot 比較像你的「開發副駕駛」，OpenClaw 則是你準備部署的 AI 助理平台。

## 三、開始養小龍蝦：安裝 OpenClaw

### 3.1 安裝 OpenClaw

最基本的安裝指令如下：

```bash
npm install -g openclaw@latest
```

安裝完成後先確認版本：

```bash
openclaw --version
```

如果你的系統找不到 `openclaw`，常見原因是：

1. Node.js 沒有安裝成功
2. npm 全域安裝路徑沒有加入 PATH
3. 目前 PowerShell 視窗尚未重新開啟

### 3.2 第一次啟動

官方推薦的第一次啟動方式是：

```bash
openclaw onboard --install-daemon
```

這個指令會啟動 Onboard 流程，協助你完成：

1. 基本設定
2. Gateway 啟用
3. Workspace 初始化
4. 模型與通道設定
5. 技能與工作環境建立

完成後，常見的啟動與操作指令如下：

```bash
openclaw gateway --port 18789 --verbose
openclaw dashboard
```

預設情況下，Dashboard 會開在：

```text
http://127.0.0.1:18789/
```

### 3.3 Windows 原生輔助安裝流程

如果你是在這份教材對應的 Windows 教學環境中操作，工作區裡另外提供了一套原生 Windows 輔助工具，特別適合剛入門、不想先碰太多命令列的使用者。

可用方式：

```powershell
.\run_openclaw_manager.ps1
```

這個工具會協助你：

1. 檢查 PowerShell、Node.js、npm、Git、Python、OpenClaw 是否已安裝
2. 自動安裝或更新依賴
3. 安裝或反安裝 OpenClaw
4. 啟動 `openclaw onboard --install-daemon`
5. 啟動與停止 Gateway
6. 直接開啟 Dashboard
7. 執行 `openclaw doctor` 做快速診斷

如果你是課堂教學現場，這個 GUI 方式通常比每台電腦逐一手打命令更穩定。

## 四、OpenClaw 的基本使用

### 4.1 指令介紹

先記住幾個最常用的 CLI 指令：

```bash
openclaw --version
openclaw status
openclaw doctor
openclaw dashboard
openclaw gateway --port 18789 --verbose
openclaw message send --target +1234567890 --message "Hello from OpenClaw"
openclaw sessions --json
```

這幾個指令的用途如下：

- `openclaw status`：看 OpenClaw 目前整體狀態
- `openclaw doctor`：做診斷，找出設定或環境問題
- `openclaw dashboard`：打開控制面板
- `openclaw gateway ...`：直接啟動 Gateway
- `openclaw message send ...`：送出一則測試訊息
- `openclaw sessions --json`：查看 session 狀況

除了 CLI，OpenClaw 在聊天介面裡也有一批常見指令：

```text
/status
/new
/reset
/compact
/think high
/verbose on
/trace on
```

你可以把它理解成：

- CLI 指令是「管理系統」
- 聊天指令是「控制目前對話」

### 4.2 基本操作流程

一個最標準的使用流程如下：

1. 安裝 OpenClaw
2. 執行 `openclaw onboard --install-daemon`
3. 啟動 Gateway
4. 打開 Dashboard
5. 接上你要用的通道或直接在 WebChat 測試
6. 用簡單任務開始，例如摘要、整理、待辦清單、查詢資料
7. 再逐步增加 Skill、排程與自動化

第一次練習可以直接丟給 OpenClaw 這種任務：

```text
請幫我整理今天的待辦事項，並依照優先順序列出。
```

接著再嘗試：

```text
請幫我規劃一份今天下午 2 點到 6 點的工作排程。
```

如果你已經接好 Telegram、Discord 或其他通道，也可以從手機或通訊軟體直接跟它互動。

### 4.3 實際應用案例

#### 案例 1：個人工作助理

你可以把 OpenClaw 當成每日工作助理，做這些事：

- 統整待辦事項
- 彙整會議紀錄
- 協助發想內容草稿
- 追蹤日常例行事項

#### 案例 2：跨平台訊息中樞

如果你同時使用 Telegram、Discord、Slack 或 Google Chat，OpenClaw 可以變成統一入口，讓你不必在不同平台之間反覆切換。

#### 案例 3：自動化報告機器人

當你後面加上 cron、heartbeat 與 Google 相關 skill 之後，OpenClaw 可以每天固定幫你：

- 做晨報
- 查郵件
- 彙整 Google Calendar 行程
- 發送摘要到指定頻道

### 4.4 OpenClaw 不會動了怎麼修復？

這一段非常重要。很多人不是卡在功能，而是卡在「昨天可以，今天突然不動」。排錯建議照下面順序做。

#### 第一步：先看版本與環境

```powershell
node --version
npm --version
openclaw --version
```

如果 Node.js 太舊，OpenClaw 很可能啟不起來。

#### 第二步：看系統狀態

```bash
openclaw status
openclaw gateway status
openclaw doctor
```

如果設定檔有錯，OpenClaw 可能會拒絕啟動。這時候優先看 `doctor` 的輸出。

#### 第三步：必要時讓 doctor 自動修復

```bash
openclaw doctor --fix
```

#### 第四步：追 Gateway 與排程狀態

```bash
openclaw cron status
openclaw system heartbeat last
openclaw logs --follow
```

這一組指令特別適合排查：

- cron 為什麼沒觸發
- heartbeat 有沒有跑
- Gateway 有沒有持續存活
- 是否有 config 驗證失敗

#### 第五步：重新啟動 Gateway

如果你懷疑是程序卡住：

1. 停掉現有 Gateway
2. 重新執行 `openclaw gateway --port 18789 --verbose`
3. 再打開 Dashboard 測試

在 Windows 教學工具中，也可以直接使用「停止 Gateway」後重新啟動。

#### 常見故障原因總整理

1. Node.js 版本太舊
2. PATH 沒刷新，導致找不到 `openclaw`
3. `openclaw.json` 設定格式錯誤
4. Gateway 已啟動，但 18789 連接埠被占用或前一個程序沒停乾淨
5. Skill 或 plugin 依賴未安裝完成
6. 通道登入狀態失效

> 實務建議：不要一出問題就整個重裝。先 `status`、再 `doctor`、再 `logs --follow`，通常就能定位到問題。

## 五、gog skill 的使用與集結 Google 全家桶

### 5.1 gog skill 的介紹

在 OpenClaw 生態中，`gog` 通常是指一條偏向 **Google 整合** 的工具鏈。實務上它常和 Gmail、Google Calendar、Google 搜尋、Google Cloud Pub/Sub 等功能一起出現，也可能被某些 Skill 或 Plugin 當成底層依賴。

你可以把它理解成：

- OpenClaw 負責 AI 流程與對話
- `gog` 或對應的 Google Skill 負責把 Gmail、Calendar、Drive、Sheets 等 Google 服務接進來

要注意的是：**不同版本、不同作者的 gog / Google 相關 skill，功能不一定完全相同。**

有些只做 Gmail，
有些可以延伸到 Calendar，
有些會再搭配 Drive、Sheets 或 Google Search。

所以教學上最安全的做法是：

1. 先確認你安裝的是哪一個 skill
2. 看它要求哪些帳號、API 金鑰或 CLI
3. 再依照該 skill 的說明做授權

### 5.2 gog skill 的使用方法

建議的操作流程如下。

#### 方法一：從 ClawHub 搜尋並安裝 Skill

先到 ClawHub 找 Google 相關 Skill，例如搜尋：

- `gog`
- `gmail`
- `google`
- `calendar`

找到後用 OpenClaw 安裝：

```bash
openclaw skills install <skill-slug>
```

更新已安裝的 skill：

```bash
openclaw skills update --all
```

#### 方法二：確認 Skill 放置位置

OpenClaw 會從多個位置讀取 Skill，其中最常用的是：

```text
<workspace>/skills
<workspace>/.agents/skills
~/.agents/skills
~/.openclaw/skills
```

如果你把 skill 裝在目前工作區，通常會出現在：

```text
skills/<skill-name>/SKILL.md
```

#### 方法三：配置 Google 相關授權

依 skill 的需求不同，常見需要準備：

1. Google 帳號授權
2. OAuth Client
3. API Key
4. `gcloud` CLI
5. 某些情況下的 `gog` CLI

例如 Gmail Pub/Sub 整合流程中，官方文件就會用到：

```bash
gcloud auth login
gcloud config set project <project-id>
gcloud services enable gmail.googleapis.com pubsub.googleapis.com
```

以及：

```bash
gog gmail watch start --account your@gmail.com --label INBOX --topic projects/<project-id>/topics/gog-gmail-watch
```

#### 方法四：直接在 OpenClaw 中呼叫工作

當 skill 已經可用時，就可以直接用自然語言下任務，例如：

```text
請幫我檢查 Gmail 未讀信，找出需要今天回覆的內容並摘要。
```

或：

```text
請幫我整理今天 Google Calendar 的行程，輸出成上午、下午、晚上三段。
```

### 5.3 gog skill 的實際應用案例

#### 案例 1：Gmail 郵件摘要助理

每天早上固定掃描郵箱，把重要信件摘要到 Telegram 或 Dashboard。

#### 案例 2：Calendar 行程助理

早上 8 點自動整理今天行程，並提醒重要會議前的準備事項。

#### 案例 3：Google 全家桶工作流

一個完整的 Google 工作流可以長這樣：

1. 用 Gmail 抓未讀信
2. 用 Calendar 對照今天行程
3. 用 Google Search 補查背景資料
4. 用 Sheets 紀錄每日統計
5. 最後把摘要發回指定聊天群組

> 教學上的重點不是死背某一個 skill 名稱，而是理解 OpenClaw 如何把 Google 生態的資料流串起來。

# ⚡ 進階教學

## 六、OpenClaw 的進階功能介紹

### 6.1 多 Agent 設定

當你不想讓所有工作都塞在同一個 AI 身上，就要開始使用多 Agent。

OpenClaw 的多 Agent 觀念是：

1. 每個 Agent 都有自己的 workspace
2. 每個 Agent 都有自己的 session 與設定
3. 每個 Agent 可以綁定不同通道、不同帳號、不同人格與不同模型

快速建立新 Agent：

```bash
openclaw agents add work
openclaw agents list --bindings
```

常見的切法：

- `main`：日常對話
- `ops`：監控與告警
- `research`：查資料與產出報告
- `social`：社群發文與回覆

### 6.2 SubAgent 的使用

SubAgent 可以把它想成「AI 的分身」。主 Agent 先接到任務，再把其中一部分丟給背景分身去做，自己不用卡住等待。

常見操作指令：

```text
/subagents list
/subagents kill <id>
/subagents log <id>
/subagents info <id>
/subagents send <id> <message>
/subagents spawn <agentId> <task> [--model <model>] [--thinking <level>]
```

SubAgent 適合做的事情：

1. 長時間資料蒐集
2. 背景整理報告
3. 多來源研究任務
4. 慢速工具操作
5. 平行化處理

實務上建議：

- 一般預設用 **isolated** 模式，讓分身用乾淨上下文工作
- 只有在分身真的需要承接目前對話上下文時，才使用 **fork**
- 不要一開始就開太多 SubAgent，先控制並發量

### 6.3 Skill 的使用

Skill 是 OpenClaw 的能力說明書與操作流程包。它不是模型本身，而是教模型「遇到某種任務時，應該如何使用工具、如何判斷流程、如何執行操作」。

一個 Skill 最核心的檔案是：

```text
SKILL.md
```

最小格式通常會包含：

```yaml
---
name: your-skill-name
description: your skill description
---
```

然後在下面寫進操作說明、限制條件、執行步驟。

Skill 的好處是：

1. 把重複流程標準化
2. 讓同一套能力可以跨任務重用
3. 讓團隊能共享操作知識
4. 讓 Agent 更穩定地調用工具

### 6.4 ClawHub 的使用

ClawHub 可以把它想成 OpenClaw 的 Skill 市集與公開技能倉庫。

你可以在 ClawHub 上：

1. 搜尋 Skill
2. 安裝 Skill
3. 更新 Skill
4. 研究社群怎麼設計工作流

最常用的指令：

```bash
openclaw skills install <skill-slug>
openclaw skills update --all
```

如果你是技能作者，還可能會用到：

```bash
clawhub sync --all
```

## 七、cron 設定與自動化

### 7.1 cron 的基本概念

cron 是 OpenClaw 的內建排程器。你可以讓它在指定時間自動喚醒 Agent 執行工作，例如：

- 每天早上 7 點生成晨報
- 每週一 6 點做專案進度分析
- 20 分鐘後提醒你回覆某封信

它和一般系統 cron 不同的地方在於：OpenClaw 的 cron 是直接長在 Gateway 裡，會保留任務定義與執行狀態。

### 7.2 在 OpenClaw 中設定 cron 任務

#### 一次性提醒

```bash
openclaw cron add \
	--name "Calendar check" \
	--at "20m" \
	--session main \
	--system-event "Next heartbeat: check calendar." \
	--wake now
```

#### 每天固定產生晨報

```bash
openclaw cron add \
	--name "Morning brief" \
	--cron "0 7 * * *" \
	--tz "America/Los_Angeles" \
	--session isolated \
	--message "Summarize overnight updates." \
	--announce
```

#### 查看排程

```bash
openclaw cron list
openclaw cron show <job-id>
openclaw cron runs --id <job-id>
```

#### 強制執行與刪除

```bash
openclaw cron run <job-id>
openclaw cron remove <job-id>
```

### 7.3 實際應用範例

#### 範例：每日營運晨報

每天早上 7 點自動：

1. 彙整前一天郵件
2. 產出今日重要行程
3. 提醒待辦事項
4. 將結果送到 Slack 或 Telegram

#### 範例：週一深度分析

每週一清晨讓 isolated session 產出較長報告，避免干擾平日主要對話。

## 八、heartbeat 設定與監控

### 8.1 heartbeat 的基本概念

如果說 cron 是「到點就做某件事」，那 heartbeat 就比較像「固定頻率巡邏，看看有沒有事情該做」。

heartbeat 的典型用途：

1. 定期檢查是否有重要待辦
2. 定期掃描 inbox 或行程
3. 定期做輕量提醒
4. 定期回報系統健康狀態

heartbeat 與 cron 的差別：

- **cron**：你明確指定工作內容與時間
- **heartbeat**：你指定巡檢節奏，讓 Agent 根據提示自己判斷有沒有要處理的事

### 8.2 在 OpenClaw 中設定 heartbeat

最基本的設定示意如下：

```json5
{
	agents: {
		defaults: {
			heartbeat: {
				every: "30m",
				target: "last",
				lightContext: true,
				isolatedSession: true
			}
		}
	}
}
```

這段設定的意思是：

1. 每 30 分鐘巡檢一次
2. 若需要通知，就送到最近一次互動對象
3. 只載入較輕量的上下文
4. 每次 heartbeat 用較乾淨的獨立 session

你也可以另外建立 `HEARTBEAT.md` 作為巡檢清單，例如：

```md
# Heartbeat checklist

- Quick scan: anything urgent in inboxes?
- Check today calendar items.
- If nothing needs attention, reply HEARTBEAT_OK.
```

### 8.3 監控 heartbeat 狀態與修復方法

查看最近 heartbeat 狀態：

```bash
openclaw system heartbeat last
```

手動喚醒一次 heartbeat：

```bash
openclaw system event --text "Check for urgent follow-ups" --mode now
```

如果 heartbeat 沒有如預期運作，優先檢查：

1. `every` 是否被設成 `0m`
2. `activeHours` 是否設定錯誤
3. `target` 是否是 `none`
4. Gateway 是否持續在線
5. `HEARTBEAT.md` 是否空白到被系統跳過

如果你想要 heartbeat 更省 token，建議：

1. 開啟 `isolatedSession: true`
2. 開啟 `lightContext: true`
3. 保持 `HEARTBEAT.md` 很短
4. 針對 heartbeat 選擇較便宜的模型

## 九、OpenClaw 的實際應用案例：如何成立一個小龍蝦軍團

當你把前面的基礎與進階功能組起來，就可以建立一套真正有分工、有自動化、有監控的 AI 團隊。

一個典型的小龍蝦軍團可以這樣配置：

### 9.1 軍團角色分工

1. **main agent**：負責日常對話、總控與回覆
2. **ops agent**：負責巡檢、heartbeat、告警
3. **research agent**：負責資料蒐集與報告
4. **content agent**：負責文案、貼文與內容整理
5. **google agent / gog skill 組**：負責 Gmail、Calendar、Sheets、Google Search 等整合

### 9.2 軍團工作流程

每天早上：

1. cron 觸發晨報
2. ops agent 檢查系統健康狀況
3. gog skill 整理 Gmail 與 Calendar
4. research agent 用 SubAgent 蒐集必要背景資訊
5. main agent 彙總後回傳給你

### 9.3 實務落地建議

如果你是第一次建立小龍蝦軍團，不要一開始就把全部功能打開。建議照這個順序長大：

1. 先讓單一 Agent 穩定啟動
2. 再加一個 Skill
3. 再加 cron
4. 再加 heartbeat
5. 最後才切多 Agent 與 SubAgent

這樣的好處是：

- 問題比較容易定位
- 成本比較容易控制
- 每一層能力都能逐步驗證

## 十、總結

如果你要把今天整份教學濃縮成一句話，那就是：

**OpenClaw 的真正價值，不只是把 AI 裝起來，而是把 AI 變成可長期運作、可分工、可監控、可自動化的工作系統。**

請先學會三件事：

1. 能安裝
2. 能啟動
3. 能排錯

接著再逐步往這三件事前進：

1. 能加 Skill
2. 能做自動化
3. 能做多 Agent 分工

當你把 `Skill + cron + heartbeat + SubAgent + Multi-Agent` 串起來時，你就不只是養一隻小龍蝦，而是在建立一支真正能工作的 AI 軍團。

## D. 使用提醒

1. `openclaw ...` 指令是在終端機打，不是在聊天框打。
2. `/...` 指令是在 OpenClaw 的聊天介面裡打，不是在 PowerShell 打。
3. `/model`、`/think`、`/trace`、`/usage` 這類指令很適合示範「同一個 agent 如何切換工作模式」。
4. `/config`、`/mcp`、`/plugins`、`/debug` 這類管理指令通常偏進階，而且很多情況預設是關閉的，不建議一開始就教初學者。
